# Demo Script

This script is written for GitHub reviewers, technical interviewers, and graduate admission interviews. It assumes the Docker stack is running and the seed scenarios are available.

## 3-minute Demo

### 0:00-0:30 Landing Page

- Open: `http://localhost:3000/`
- Show: project positioning, system-engineering value, and stack summary.
- Say: "This is a civil emergency resource scheduling platform. It models road networks, depots, vehicles, demands, congestion, and blocked roads, then compares dispatch algorithms through reproducible metrics."

### 0:30-1:00 Scenario Management

- Click: `Scenarios` or open `/scenarios`.
- Show: six seeded scenarios and counts for nodes, edges, vehicles, and demands.
- Say: "The project is not a single static demo. It includes benchmark scenarios for normal traffic, congestion, demand surge, resource shortage, blocked roads, and high-priority rescue tasks."

### 1:00-1:40 Network Visualization

- Open: `/network`.
- Show: depots, vehicles, demands, priority markers, congested edges, and blocked edges.
- Say: "The road network is modeled as a weighted directed graph. Blocked edges are excluded from routing; congestion increases travel-time cost."

### 1:40-2:20 Simulation Control

- Open: `/simulation`.
- Select: one scenario, then click `Compare Algorithms`.
- Show: Greedy, Priority, Dijkstra, A*, and OR-Tools CVRP comparison running through the same API path.
- Say: "The same scenario can be evaluated by multiple algorithms under the same metrics, which makes the result comparable."

### 2:20-3:00 Results and Report

- Open: `/results`, then `/report`.
- Show: metric cards, charts, weighted score ranking, and generated Markdown report.
- Say: "The output is persisted as simulation runs, dispatch plans, routes, and metric results. The report is deterministic, so it works without an LLM API key."

## 5-minute Demo

### 0:00-0:45 Project Positioning

- Open the README first if presenting from GitHub.
- Show badges, screenshot gallery, quick start, architecture, and algorithm flow.
- Say: "The project is designed so a reviewer can understand the value in three minutes, then inspect the code and docs for details."

### 0:45-1:30 Data Model and Scenarios

- Open `/scenarios`.
- Choose `Blocked Road Scenario` or `High Priority Rescue Scenario`.
- Say: "Each scenario is generated by seed data rather than hard-coded UI state. Scenario summaries expose the experimental challenge."

### 1:30-2:15 Graph Visualization

- Open `/network?scenarioId=<id>` if available through navigation.
- Hover over graph elements.
- Say: "This is where data structures become visible: nodes and edges represent the graph, while depots, vehicles, and demands are domain objects."

### 2:15-3:15 Algorithm Comparison

- Open `/simulation`.
- Run `Compare Algorithms`.
- Say: "Greedy is a fast baseline, Priority emphasizes urgent tasks, Dijkstra and A* show shortest-path planning, and OR-Tools CVRP is the operations research baseline for capacity constraints."

### 3:15-4:15 Metrics Dashboard

- Open `/results`.
- Show total distance, response time, completion rate, priority completion, runtime charts, and weighted score.
- Say: "Emergency scheduling cannot be judged only by distance. Completion rate and priority completion have higher weights because urgent service matters most."

### 4:15-5:00 Report

- Open `/report`.
- Generate a report with the latest run id.
- Say: "The report explains scenario description, algorithm configuration, metric comparison, bottlenecks, recommendations, limitations, and future work."

## 10-minute Graduate Interview Walkthrough

### 0:00-1:00 Motivation

- Show README title and project summary.
- Say: "I chose civil emergency scheduling because it combines system modeling, path planning, operations research, simulation evaluation, and engineering implementation."

### 1:00-2:00 System Architecture

- Show README Mermaid architecture.
- Explain: frontend, backend, database, algorithm services, report generator, Redis, and Celery worker.
- Emphasize: "The dashboard is not static. It consumes real FastAPI results."

### 2:00-3:00 Mathematical Modeling

- Show `docs/project_summary_3_pages.md` or `docs/algorithm_design.md`.
- Explain: `G=(V,E)`, edge distance, speed, congestion factor, blocked status, vehicle capacity, demand quantity, priority, and service time.

### 3:00-4:00 Algorithms

- Show README algorithm flow.
- Explain:
  - Dijkstra: non-negative weighted shortest path.
  - A*: adds geographic heuristic.
  - Greedy: fast local baseline.
  - Priority: urgent-task-first dispatch.
  - OR-Tools CVRP: capacity-constrained routing baseline.

### 4:00-5:00 Standard Scenarios

- Open `/scenarios`.
- Compare Normal, Congestion, Demand Surge, Resource Shortage, Blocked Road, and High Priority Rescue.
- Say: "These scenarios let me discuss robustness rather than only one easy case."

### 5:00-6:30 Live Run

- Open `/simulation`.
- Select `High Priority Rescue Scenario`.
- Click `Compare Algorithms`.
- Say: "This run produces persisted simulation records. The frontend does not invent metrics."

### 6:30-8:00 Results Interpretation

- Open `/results`.
- Discuss:
  - Lower distance is good but not sufficient.
  - Priority completion is important in emergency settings.
  - Runtime matters for operational usability.
  - Weighted score is explainable, not absolute truth.

### 8:00-9:00 Report Generation

- Open `/report`.
- Generate Markdown.
- Say: "Without an API key, the deterministic template still produces a readable experiment report. With an OpenAI-compatible key, it can polish the language."

### 9:00-10:00 Limitations and Future Work

- Show Roadmap.
- Say: "Current limitations are static scenarios, partial time-window evaluation, grid-based roads, and no PDF export. Next I would implement rolling-horizon re-optimization, full VRPTW, and real map integration."
